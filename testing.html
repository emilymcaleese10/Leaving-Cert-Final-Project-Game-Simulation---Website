Testing


Extent of Testing

For the majority of my project, I used functional testing to ensure tasks were performed as expected. As I added features to enhance the basic model eg. scoreboard, I used unit testing to ensure that it functioned as intended. Once the function’s expected outcomes were the same as the actual outcomes, I used integration testing to test whether the function would work with other parts of the program. Then, I used system testing to test how everything worked together. I used acceptance testing to ensure I fulfilled all project requirements and objectives and I used usability testing by showing my artefact to friends to ensure the game was navigable and had a friendly user-interface.

Problems

After creating a basic model of the game that only allowed the user and computer make moves, I tested its functioning. When the computer chose a square that had already been filled by the user’s X, it would be overwritten with the computer’s O. To fix this, I added an if statement into the update_grid() function so that the move would only be made if the chosen square was empty.

The print_menu() function returns the user input from the menu. After placing it in an if-statement for each game mode, the menu printed 3 times. I used unit testing to ensure the problem was not in the function itself. Then, I employed integration testing by running the if-statement with the function. When the problem occured, I knew it involved the if-statement. I learned that functions are run when placed in if statements, something I didn’t know previously. To fix this, I assigned the function to a variable and placed the variable in the same if-statement so it was only run once.

Once I had a working model for each game mode, I tested the user inputs with accepted inputs (5), extreme values (100), boundary values (1,9) and error anticipation (“g”,4.5, -7). My original design used an if-statement to only accept integers between 1 and 9. Using error anticipation testing, the inputs 4.5 and “g” returned an error.

Test description: Test to validate user input for making move, should only accept integers (1-9)

I changed my input validation design so that if tried converting the input into an integer and got an error, it would now print “Try again” and prompt user for another input. Then all test data passed.
